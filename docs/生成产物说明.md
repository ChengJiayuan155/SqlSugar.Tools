# 代码生成：生成产物说明（从大到小）

本说明文档用于回答两个问题：

1. **到底生成了哪些层/哪些目录/哪些文件？**
2. **每个文件有什么用？放进项目后怎么快速跑起来？**

> 说明：本工具的目标是对齐“模板驱动生成器”的体验 —— **只需要写控制器/业务逻辑，其它基础设施（目录结构、基础类、统一返回体、权限骨架、DI 注册）尽量自动补齐**。

---

## 一、生成输出根目录

你选择输出目录后，生成器会在该目录下创建多个“项目目录”，**目录名就是命名空间**：

- `{RootNamespacePrefix}.{ProjectName}.Model`
- `{RootNamespacePrefix}.{ProjectName}.DAL`
- `{RootNamespacePrefix}.{ProjectName}.BLL`
- `{RootNamespacePrefix}.{ProjectName}.Common`
- `{RootNamespacePrefix}.{ProjectName}.WebApi`

例如你填写：

- RootNamespacePrefix = `QingQia`
- ProjectName = `Smartcat`

那么会生成：

- `QingQia.Smartcat.Model`
- `QingQia.Smartcat.DAL`
- `QingQia.Smartcat.BLL`
- `QingQia.Smartcat.Common`
- `QingQia.Smartcat.WebApi`

---

## 二、Model 层（最重要：生成与扩展分离）

### 1) `generat/<表名>/Model<表名>.cs`

用途：**表映射实体**（数据库字段一一对应）。

特点：

- 类为 `public partial class Model<表名>`
- 包含：
  - `[SugarTable("原表名", TableDescription="表注释")]`
  - 每个字段的 `[SugarColumn(ColumnDescription=..., IsPrimaryKey/IsIdentity/IsNullable)]`
- 只包含数据库真实列（不在这里放扩展字段）

为什么这么做：

- 生成文件可能被反复覆盖更新
- 不希望你写的扩展字段被覆盖掉

### 2) `temp/<表名>/Model<表名>_partial.cs`

用途：**扩展部分类**（你自己写）。

建议放：

- 导航属性（`[Navigate]`）
- 非表字段（`[SugarColumn(IsIgnore=true)]`）
- 枚举/字典映射字段
- 业务展示字段

### 3) `temp/<表名>/Model<表名>_search.cs`

用途：**后台搜索类**（用于筛选/分页查询参数）。

建议放：

- `xxx_start/xxx_end`（时间区间）
- `keyword`（关键词）
- 各种 where 条件字段

---

## 三、Common 层（依赖倒置 + SqlSugarScope）

### 1) `ISqlSugarDbContext.cs`

用途：上层只依赖接口，便于替换/测试。

### 2) `SqlSugarDbContext.cs`

用途：包装 `SqlSugarScope`，用于注入到 DAL/Repository。

---

## 四、DAL 层（Repository 封装）

### 1) `IRepository<T>.cs`

用途：统一 CRUD 能力（同步 + 异步）。

包含：

- Queryable()
- GetById / GetList
- Insert / InsertBatch
- Update / UpdateColumns
- DeleteById / Delete(where)

### 2) `Repository<T>.cs`

用途：对 `SqlSugarScope` 的具体调用封装。

---

## 五、BLL 层（生成与扩展分离）

### 1) `BaseBll.cs`

用途：对 Repository 做“业务层常用封装”，让控制器更轻。

### 2) `generat/<表名>/Bll<表名>.cs`

用途：生成的业务类壳子（可反复生成覆盖）。

### 3) `temp/<表名>/Bll<表名>_partial.cs`

用途：扩展业务逻辑（你自己写，不会被覆盖）。

---

## 六、WebApi 层（统一返回体 + 权限骨架 + 可直接调用）

### 1) `_base` 基础控制器链

- `_BaseApiController.cs`：最底层 API 控制器
- `BaseApiController.cs`：二层基类
- `BaseController.cs`：常用基类，内置：
  - `SuccessString() / ErrorString()` 统一返回

### 2) `_base/Results` 统一返回体（替代 Ok()）

- `SysResult.cs`：包含
  - `SysResult`（`code/msg/data`）
  - `SysResultString`
  - `SysResult_layui_table`（分页 table 返回）
  - `PagedResult<T>`（分页数据容器）
  - `formparam_PagerInfo_LayUI`（分页入参）
  - `formbody_param_del`（批量删除入参）
  - `formbody_param_switch`（开关入参）

约定：

- `code == 0` 表示成功
- `msg` 是提示信息
- `data` 是数据体

### 3) `_base/HelperPowers` 权限骨架

- `PowerMenu.cs`：全局入口 + 常用权限动作枚举 + 占位特性
- `PowerMenu_<表名>.cs`：每表一个权限常量文件

说明：

- 这里的特性/权限逻辑是“可编译占位版”
- 宿主项目有自己的鉴权体系时，可替换/对接

### 4) `Services` 服务注册

- `Services/ServiceCollectionExtensions.cs`
- 提供 `AddGeneratedServices()`，将 `Bll<表名>` 注入 DI

### 5) `Controllers` 控制器

每表一个 Controller，默认生成方法（与模板生成器风格对齐）：

- `get_page(...)`：分页列表（返回 `SysResult_layui_table`）
- `do_add(...)`：新增（返回 `SysResultString`）
- `do_edit(...)`：编辑（返回 `SysResultString`）
- `do_delete(...)`：删除（返回 `SysResultString`）

> 注意：`get_page` 的 where/join/includes/主键删除等，需要按业务补齐（生成器先给结构和返回体，保证项目整体跑起来）。

---

## 七、把生成结果放进项目后怎么“跑起来”

最短路径：

1. 把生成出来的五个目录（Model/DAL/BLL/Common/WebApi）加入解决方案（可以新建对应项目，也可以直接放到现有项目目录）。
2. 在 WebApi 启动处（`Program.cs` / `Startup.cs`）注册：

- `services.AddGeneratedServices();`

1. 配置 SqlSugarScope / ConnectionConfig（由你的宿主项目负责）。

---

## 八、常见调整点（推荐实践）

- **扩展字段一定写在 Model temp 的 partial**，不要写在 generat。
- **业务逻辑写在 BLL temp 的 partial**，不要写在 generat。
- **统一返回体可按你的前端规范扩展字段**：
  - `traceId`
  - `timestamp`
  - `errorCode`
  - `errors`（字段级错误）
